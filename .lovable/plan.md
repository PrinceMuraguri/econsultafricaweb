

# Enable Google Sign-In + Simplify Registration Flow

## Problem
The current registration form is too long (name, username, email, password, sex, age, phone, country, occupation, interests, terms). This creates friction and discourages signups. Users should be able to join with one click via Google.

## Plan

### 1. Configure Google OAuth (tool-driven)
Run the **Configure Social Auth** tool to generate the `src/integrations/lovable` module with managed Google OAuth support. This handles all OAuth infrastructure automatically — no API keys needed.

### 2. Add Google Sign-In to LoginModal and RegistrationModal
- Add a "Continue with Google" button at the top of both modals, above the email/password form
- Use `lovable.auth.signInWithOAuth("google", { redirect_uri: window.location.origin })` 
- Separate with an "or" divider between Google button and the email form
- After Google sign-in, the `handle_new_user` trigger auto-creates a `user_profiles` row using metadata from the Google account (name, email-derived username)

### 3. Simplify the Registration Form
Strip the registration form down to essential fields only:
- **Keep**: Full Name, Username, Email, Password, Terms checkbox
- **Remove from signup**: Sex, Age Bracket, Phone, Country, Occupation, Organization, Interests

These removed fields become optional profile fields the user can fill in later on their profile page.

### 4. Collect Phone Number at Capital Commitment
When a user tries to commit capital (stake on Pro Arena) and their profile has no phone number:
- Show a quick phone number collection modal before proceeding to checkout
- Save to `user_profiles.phone`
- This is the natural point where phone numbers matter (for M-Pesa payouts)

### 5. Auth Trigger on Poll Click
The current flow already opens the RegistrationModal when an unauthenticated user clicks to vote — this stays the same. The difference is the modal will now have Google Sign-In prominently at the top, making it a one-click action.

## Technical Detail
- The `handle_new_user()` database trigger already creates `user_profiles` on signup using `raw_user_meta_data`. For Google OAuth users, this pulls `full_name` from Google and derives `username` from the email prefix.
- Phone number will be nullable (it already is in the schema), so no migration needed.
- The phone collection modal will be a lightweight component shown in `StakeModal` / `TradingPanel` before proceeding with Paystack checkout.
